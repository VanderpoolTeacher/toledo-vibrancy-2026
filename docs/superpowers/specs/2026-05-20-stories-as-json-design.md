# Stories-as-JSON — design

**Date:** 2026-05-20
**Scope:** Refactor the dashboard's data model from "hand-written HTML page rebuilt daily" to "structured story corpus + per-day editorial index, rendered client-side." Stories become persistent, query-able JSON entries. The dashboard view is computed from them. The daily 5 AM task writes JSON, not HTML.

## Goal

Today, every story lives as an HTML block inside `dashboard/index.html`. The daily 5 AM task rewrites that file each morning. There is no structured corpus, no easy way to query across days, and no clean way for stories to evolve over time.

This refactor makes the **story** a first-class persistent record. The dashboard becomes a renderer over that record set, parameterised by date. The daily task's job simplifies to "write the right JSON files."

The motivating concerns:

- **10–25 net-new stories per day** (per user). The current HTML-rewrite model doesn't scale past a few weeks.
- **Find connections between stories.** Tags, people, places, related-story links become queryable.
- **Past-day views.** Click "May 19" and see that day's dashboard, not a markdown file on GitHub.
- **Editorial continuity.** A story that recurs over weeks (e.g., Huron Yards) should have one canonical record that evolves, not duplicated text every day.

## Non-goals

- A query UI / search across stories. The data model enables it; the UI is future work.
- Backfilling pre-cutover dates. May 19's existing markdown stays as the human-readable record; we don't retroactively parse it into JSON.
- A backend / database. Static files on GitHub Pages.
- User accounts / cross-device persistence for follows. Separate concern.

## Architecture

Single static page on GitHub Pages, no build step, no framework — same constraints as today.

### File layout

```
dashboard/
  index.html                                ← thin shell: CSS + render JS, no story HTML
  stories/
    <id>.json                               ← canonical story (one file per story, ever)
    swanky-scoops.json
    huron-yards-phase-two.json
    watershed-weekend-riverwalk.json
    ...
  days/
    2026-05-20.json                         ← per-day index
    2026-05-19.json
    ...
  archive/
    2026-05-20_toledo_development_brief.md  ← auto-generated from days/ + stories/
    2026-05-19_toledo_development_brief.md  ← pre-cutover; unchanged
    ...
  scripts/
    render_brief.py                         ← day-file + stories → markdown brief
    extract_today.py                        ← ONE-TIME bootstrap; can be retired post-cutover
  followed_stories.json                     ← unchanged
  coords-source.md                          ← unchanged (audit trail for story coordinates)
```

### Three roles, three formats

1. **`stories/<id>.json`** — canonical record. Exists once per story. Edited in place as the story evolves.
2. **`days/<date>.json`** — the editorial decision for one day: which stories appear, in what sections, what order, which is the top story, which carry the "ongoing" treatment, the watch-list, the live-sources.
3. **`archive/<date>_toledo_development_brief.md`** — a derived artifact, auto-generated by `render_brief.py`. Readable narrative, used for human reading and LLM context.

JSON is the source of truth. Markdown is a derived view.

## Story JSON schema

```json
{
  "id": "swanky-scoops",
  "headline": "Swanky Scoops opening a Warehouse District scoop shop",
  "section": "Warehouse District",
  "tags": ["warehouse"],

  "posted_date": "2026-05-20",
  "source_date": "2026-05-16",
  "last_updated": "2026-05-20",

  "paragraphs": [
    "The Blade reported (May 16) that Alissa Vieira's artisanal ice cream business..."
  ],
  "why_it_matters": "Small ticket on its own, but it lands in the right place...",

  "people": [
    {
      "name": "Alissa Vieira",
      "role": "Owner, Swanky Scoops",
      "search_terms": "Alissa Vieira Swanky Scoops Toledo"
    }
  ],

  "sources": [
    { "title": "The Blade", "url": "https://www.toledoblade.com/a-e/food/..." }
  ],

  "locations": [
    {
      "shape": "point",
      "label": "Warehouse District (block TBD)",
      "match": null,
      "lat": 41.6493,
      "lon": -83.5407
    }
  ],

  "external_map_link": null,

  "related_story_ids": ["huron-yards-phase-two"],

  "search_terms": "Swanky Scoops Warehouse District Toledo Vieira ice cream opening"
}
```

For multi-site stories (e.g., ConnecToledo's 4 parks), `locations` carries multiple `point` entries. For stories that link out to a separate map instead of an inline mini-map (the original vibrancy-19-grants top story), set `locations: []` and `external_map_link: { href, label }`.

### Field reference

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | string (kebab-case slug) | yes | matches existing `data-story-id` |
| `headline` | string | yes | rendered as `<h2>`/`<h3>` |
| `section` | string | yes | plain string; e.g., "Warehouse District" |
| `tags` | string[] | yes | values: `downtown` \| `warehouse` \| `vistula` \| `uptown` \| `junction` \| `grants` \| `citywide` |
| `posted_date` | `YYYY-MM-DD` | yes | when first added to the dashboard |
| `source_date` | `YYYY-MM-DD` \| `""` | yes (may be empty) | when the underlying news happened |
| `last_updated` | `YYYY-MM-DD` | yes | most recent daily-task touch |
| `paragraphs` | string[] | yes (≥1) | story body; rendered as `<p>` |
| `why_it_matters` | string | yes | rendered in the existing `.why` block |
| `people` | object[] | yes (may be `[]`) | each: `{name, role, search_terms}` |
| `sources` | object[] | yes (may be `[]`) | each: `{title, url}` |
| `locations` | object[] | yes (may be `[]`) | 0..N location entries; drives the story-map feature |
| `locations[].shape` | `"point"` \| `"line"` \| `"polygon"` | when entry present | |
| `locations[].label` | string | when entry present | popup tooltip / chip text |
| `locations[].match` | string \| null | when entry present | substring to wrap in prose; `null` → headline chip |
| `locations[].lat`/`lon` | number | when shape is `point` | |
| `locations[].path` | `[[lat,lon], ...]` | when shape is `line` | |
| `locations[].bbox` | `[[lat,lon], [lat,lon]]` | when shape is `polygon` | |
| `external_map_link` | object \| null | optional | `{href, label}` for stories that link to an external map instead of rendering inline |
| `related_story_ids` | string[] | optional | seeds future "connections" UI |
| `search_terms` | string | yes (may be empty) | matches existing `data-search-terms` |

### What's NOT on the story

- `is_top_story` / `is_ongoing` — these are per-day visual treatments, not properties of a story. Live in the day-file.
- `featured` / `priority` — editorial decisions, also per-day.

```json
    { "title": "Warehouse District", "story_ids": ["swanky-scoops", "huron-yards-phase-two", "port-warehouse-liquid-terminal"] },

    { "title": "Uptown / Junction", "story_ids": ["innovation-post-tenants", "raise-uptown-junction"] },

    "huron-yards-phase-two", "port-warehouse-liquid-terminal", "vistula-metropark", "soccer-stadium-vistula",
    "innovation-post-tenants", "raise-uptown-junction", "glass-city-urban-campground", "owens-corning-250m",
    "connectoledo-music-grant", "small-business-app-growth", "promedica-medical-mutual"

    "Vibrancy 19 — property list.",

  ],

    { "title": "City of Toledo news", "url": "https://toledo.oh.gov/news" },

```

### Day-file rules

- **Carry-forward is explicit.** Every story shown today must appear in today's `sections`. There's no automatic "active stories" derivation; the daily task lists each one.
- **`top_story_id` must reference a story_id that appears in `sections`.** Otherwise it's a dangling reference.
- **`ongoing_story_ids` is a subset of all `story_ids` in `sections`.** Stories in this list get the gray `is-ongoing` background; others render with the default white card.
- **`watch_list` and `live_sources` move out of `index.html` into the day-file** so they can change day-over-day.

## Render pipeline

`dashboard/index.html` becomes a thin shell. Layout:

```
<header>...</header>
<div class="banner">...</div>
<div class="follow-panel">...</div>
<div class="past-briefs" id="past-briefs"></div>     ← populated at render time
<div class="filters" id="filters">...</div>           ← static; no change

<div id="loading-banner">Loading stories…</div>       ← shown while fetching
<main id="stories"></main>                            ← cards rendered here

<div class="watch-list" id="watch-list"></div>        ← populated at render time
<div class="live-sources" id="live-sources"></div>    ← populated at render time
<div class="footer-note">...</div>
```

### Load sequence

1. Parse target date from `?date=YYYY-MM-DD` URL param, default to today (local TZ).
2. `fetch('days/<date>.json')` → day-file. On 404, render "No brief published for `<date>`." and stop.
3. Collect every unique `story_id` from `day.sections`.
4. `Promise.all(story_ids.map(id => fetch('stories/<id>.json')))` — parallel fetches. On 404 for an individual story, log + render an error placeholder card for that ID; continue with the rest.
5. Render header bits from day-file (title, window, auto_refresh in the meta strip).
6. Render past-briefs pills: one pill per `?date=YYYY-MM-DD` link for the previous 1–3 days that have a corresponding `days/<date>.json` (detected via 404-fast probe, or simpler: just attempt yesterday's date and let the click 404-handle if missing), plus a static "All →" link to `archive/`.
7. For each section in `day.sections`: render a `.section-title` then each story card in order. Apply `.top-story` class if `id === day.top_story_id`; apply `.is-ongoing` class if `day.ongoing_story_ids.includes(id)`.
8. Render watch-list from `day.watch_list`.
9. Render live-sources grid from `day.live_sources`.
10. **After cards are in the DOM**, run the existing init pipeline in order: `loadAuthorFollows()`, `renderAuthorBadges()`, `renderPersonChips()`, `renderStoryFollowState()`, `renderFollowPanel()`, `hydrateStoryTriggers()`, `applyFilter()`. These functions all walk the DOM, so they must run after rendering, not before.
11. Hide loading banner.

### Story-card rendering

The card DOM exactly mirrors today's structure (same classes, same `data-` attributes), so the existing follow-panel, filter, person-chip, and story-map JS all work unchanged. Specifically:

- `data-story-id`, `data-tags`, `data-people` (pipe/semicolon-encoded), `data-source-date`, `data-search-terms` — same shape as today, populated from JSON fields.
- The `STORY_LOCATIONS` JS object is removed. The story-map feature now reads `story.location` from each fetched story; the same `wrapMatchInCard`, `appendHeadlineChip`, `openMapFor`, `renderItem` flow applies.

### Date routing

- `dashboard/` → today.
- `dashboard/?date=2026-05-19` → that specific day.
- The "Past briefs" pill links are `<a href="?date=2026-05-19">May 19</a>` — same page, different data.

### Filter, follow, story-map JS

Unchanged in behavior. The render pipeline produces the same DOM shape today's hand-written HTML produces, so:

- Filter pills hide cards by `data-tags` ↔ filter value (no change).
- Follow panel localStorage keys, person chips, author follows from `followed_stories.json` (no change).
- Story-maps lazy-init per card (no change, except the data source moves to `story.location`).

## Daily task contract

The 5 AM scheduled agent's new responsibilities, in order:

1. **Read existing state.** List `dashboard/stories/` to avoid ID collisions. Read yesterday's `dashboard/days/<yesterday>.json` to know what was relevant yesterday.
2. **For each NEW story discovered today:** write `dashboard/stories/<new-id>.json` with `posted_date: <today>`, `last_updated: <today>`, full content.
3. **For each EXISTING story with new developments:** edit `dashboard/stories/<id>.json` in place — update paragraphs, add sources, refresh `last_updated`. **Do NOT change `posted_date`.**
4. **Write `dashboard/days/<today>.json`** — top_story_id, sections + ordering, ongoing_story_ids, watch_list, live_sources.
5. **Run `python3 dashboard/scripts/render_brief.py <today>`** → writes `dashboard/archive/<today>_toledo_development_brief.md`.
6. **Run `dashboard/deploy.sh`** (unchanged) — stages `dashboard/`, commits, pushes.

### Idempotency

- Re-running for the same date overwrites `days/<date>.json` and the archive markdown.
- New story IDs must be checked against existing files before creation.
- `last_updated` bumps on each touch.

### Pre-commit validation

`deploy.sh` gets a one-line addition:

```sh
jq empty dashboard/stories/*.json dashboard/days/*.json dashboard/followed_stories.json
```

If any file has invalid JSON, the push fails loudly.

## Markdown renderer

`dashboard/scripts/render_brief.py` is small (~80 lines of Python). Reads `dashboard/days/<date>.json`, then for each `story_id` in sections, reads `dashboard/stories/<id>.json`. Emits markdown matching the existing format:

- Title + date header
- Top story section
- One section per `day.sections` entry, with `## <section title>` and `### <headline>` per story
- Paragraphs, "Why it matters", source links
- Watch list section
- Sources index

Output goes to `dashboard/archive/<date>_toledo_development_brief.md`.

## Migration plan (one-time cutover)

### Step 1 — Extract today's HTML into JSON

`dashboard/scripts/extract_today.py` parses `dashboard/index.html` and the existing inline `STORY_LOCATIONS` JS object. For each `.story-item`, emits `dashboard/stories/<id>.json` populated from the HTML attributes and content. Builds `dashboard/days/2026-05-20.json` from the section structure and today's `.top-story` / `.is-ongoing` classes.

- All extracted stories get `posted_date: "2026-05-20"` (cutover date). We don't pretend to know prior posting history.
- The extraction script is one-time; archived under `scripts/` for reference but not part of the normal pipeline.

### Step 2 — Markdown renderer

Build `scripts/render_brief.py`. Run it for `2026-05-20`, verify output against the existing `dashboard/archive/2026-05-20_toledo_development_brief.md`. Replace the existing markdown with the rendered version. Diffs in prose phrasing are acceptable; structural equivalence (sections, stories, sources) is required.

### Step 3 — New `dashboard/index.html` shell

Replace body content with the empty shell described under Render pipeline. Keep all existing CSS verbatim. Refactor the existing JS:

- Remove the per-story HTML (lines 361–610-ish).
- Remove the `STORY_LOCATIONS` constant; location data now lives on each story.
- Replace the static `init()` flow with the load-sequence above.
- Story-map functions, follow functions, filter functions: unchanged.

### Step 4 — Atomic cutover commit

A single commit containing:

- New `dashboard/stories/*.json` and `dashboard/days/2026-05-20.json` files
- Replaced `dashboard/index.html` (the new shell)
- New `dashboard/scripts/render_brief.py`
- New (one-time) `dashboard/scripts/extract_today.py`
- Updated `dashboard/deploy.sh` (with `jq` pre-commit validation)
- Regenerated `dashboard/archive/2026-05-20_toledo_development_brief.md`

Revert this single commit if anything breaks post-deploy.

### Step 5 — Update the daily-task prompt

Once the cutover is verified live, paste the new daily-task contract (from the Daily task contract section above) into the scheduled agent via `/schedule`. This step is manual — the agent's prompt lives outside this repo.

### Step 6 — Optional, deferred: retroactive May 19 backfill

Don't do this now. The May 19 markdown brief stays as the human-readable record. `dashboard/?date=2026-05-19` will return "No brief published for 2026-05-19" which is honest given we never had structured data for that day.

## Error handling

| Failure | Behavior |
|---|---|
| `days/<date>.json` 404 | Render "No brief published for `<date>`." in the cards area; suppress watch-list and live-sources. |
| Individual `stories/<id>.json` 404 | Render a single error placeholder card with the story ID; continue rendering the rest. |
| All fetches fail (offline) | Render a top-of-page banner: "Couldn't load today's brief — check your connection." |
| Leaflet CDN failure (existing behavior) | Story-map clicks fall through to Google Maps; preserved from current story-maps feature. |
| Invalid JSON committed | `jq empty` in `deploy.sh` fails the push. Manual fix required. |
| Daily task collides on a story ID | Daily-task prompt requires reading `stories/` before creating; if it happens anyway, last write wins and the original story is overwritten. Mitigation: prompt discipline. |
